# 27 — Testing Strategy

> **"If it's not tested, it's broken. You just don't know it yet. In an AI platform, testing is the only way to distinguish between a deterministic bug and a non-deterministic hallucination."**

---

# Purpose

This document outlines the testing approach for PROJECT BRAIN. Traditional testing pyramids do not perfectly map to AI-native applications because the core value is generated by non-deterministic LLMs. We implement a hybrid testing strategy: a traditional pyramid for the deterministic infrastructure, and an Evaluation Suite for the AI.

---

# 1. Deterministic Testing (The Infrastructure)

The platform surrounding the AI must be bulletproof. If the API or Database fails, the AI cannot function.

### 1.1. Unit Tests (Fast & Isolated)
- **Scope:** Individual functions, service layer logic, and utilities.
- **Framework:** `pytest` (Backend) / `Jest` (Frontend).
- **Rule:** **Never hit a real database or a real external API in a unit test.** 
- **Mocking:** Use `unittest.mock` (Python) to mock database sessions and external services.

### 1.2. Integration Tests (Moderate Speed)
- **Scope:** API endpoints checking request validation, database writes, and response formatting.
- **Framework:** `pytest` with `Testcontainers` (or an isolated Docker-composed test database).
- **Rule:** Test against a real PostgreSQL instance. Do NOT use SQLite for integration tests because SQLite does not support `JSONB` or `pgvector` accurately.
- **Mocking the LLM:** In integration tests, the call to the LLM Gateway (LiteLLM) MUST be mocked to return a deterministic JSON payload. We are testing our code's ability to parse the response, not the LLM's ability to generate it.

### 1.3. End-to-End (E2E) Tests (Slow & Comprehensive)
- **Scope:** Critical user journeys through the browser (e.g., Logging in, navigating to the Dashboard, submitting a form).
- **Framework:** `Playwright` (Preferred over Cypress due to speed and native async support).
- **Rule:** Keep these to a minimum (the "Happy Paths"). E2E tests are brittle and slow.

# 2. Non-Deterministic Testing (The AI Evaluation Suite)

You cannot write a standard assertion like `assert response == "The company is a B2B SaaS"` for an LLM output. Instead, we use an Evaluation Suite.

### 2.1. The Golden Dataset
We maintain a "Golden Dataset" of 50-100 real-world inputs (e.g., 50 different company names and their contexts) and their expected outputs (not exact text, but expected facts or JSON structures).

### 2.2. Automated Evaluation Metrics
Nightly, a separate CI job runs the AI Prompts against the Golden Dataset using real LLMs. It evaluates the outputs using a stronger "Judge LLM" (e.g., using GPT-4 to judge a Gemini 1.5 Flash output).

**We measure:**
1. **Structural Adherence:** Did the output match the requested Pydantic schema perfectly? (Pass/Fail)
2. **Factuality (Hallucination Check):** Did the LLM invent facts not present in the provided context? (Scale 0-1)
3. **Relevance:** Did it answer the specific prompt? (Scale 0-1)

If the overall score drops below a predefined threshold (e.g., 0.85), an alert is triggered in Slack. This prevents "Prompt Drift" where a seemingly innocent change to a system prompt degrades output quality across the board.

# 3. Test Driven Development (TDD) in the AI Era

When using AI Coding Agents (like Antigravity or Codex), the paradigm shifts from TDD to **Test-Verified Development (TVD)**:
1. The Human defines the Acceptance Criteria and interfaces.
2. The AI Agent writes both the implementation and the tests.
3. The Human reviews the tests to ensure they actually prove the Acceptance Criteria, then runs them.

**Crucial Rule:** Do not trust an AI agent that writes code without writing the corresponding test. An AI is highly prone to writing tests that assert tautologies (e.g., `assert True == True`). The human must review the *assertions* carefully.
