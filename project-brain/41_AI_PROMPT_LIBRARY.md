# 41 — AI Prompt Library (Mega-Prompts)

> **"Don't reinvent the wheel. Use proven, highly-structured mega-prompts to extract maximum performance from AI Coding Agents."**

---

# Purpose

This document stores highly effective, reusable **Mega-Prompts** for interacting with AI Coding Agents (like Antigravity) during the development of PROJECT BRAIN. 

A Mega-Prompt is not a simple one-liner. It is a carefully engineered prompt that injects constraints, roles, and context boundaries to prevent the LLM from hallucinating bad architecture.

---

# 1. The "Greenfield Feature" Prompt

**Use Case:** You need the AI to build a new Backend API feature from scratch (Models, Schemas, Service, Router).

**Copy & Paste this into your AI Agent:**
```text
Act as a Principal Python Engineer building a highly scalable, multi-tenant FastAPI application.

I need to create a new feature for the [INSERT MODULE NAME] module. 
Before writing any code, you MUST read the following foundational documents:
1. `22_CODING_STANDARDS.md`
2. `23_API_GUIDELINES.md`
3. `24_DATABASE_SCHEMA_GUIDE.md`
4. `26_BACKEND_GUIDELINES.md`

Based on those strict guidelines, implement the CRUD operations for [INSERT ENTITY NAME].

Requirements:
1. Generate the SQLAlchemy Model (inheriting from our Base, including `workspace_id` and soft delete columns).
2. Generate the Pydantic Schemas (Create, Update, Response).
3. Generate the Service Layer methods. Do not leak HTTP Request objects here.
4. Generate the FastAPI Router. You MUST inject `Depends(get_current_workspace)` and pass `workspace_id` to every service method to enforce tenant isolation.

Do not write the tests yet. Just write the core implementation. Ensure strict type hinting (Mypy compliant) and use the Bouncer Pattern for error handling.
```

# 2. The "Ruthless Refactor" Prompt

**Use Case:** You have messy code (or code generated by a weaker LLM) and you need it brought up to Enterprise standards.

**Copy & Paste this into your AI Agent:**
```text
Act as a strict, pedantic Staff Software Engineer reviewing code.

Please review the file `[INSERT FILE PATH]`.
Refactor this code to strictly adhere to our Coding Standards defined in `22_CODING_STANDARDS.md`.

Specifically, you must ruthlessly eliminate the following "Code Smells":
- Magic strings or numbers (extract them to Enums or Constants).
- Swallowed exceptions (no `except Exception: pass`).
- Missing or `Any` type hints.
- Nested if/else blocks (flatten them using the Bouncer Pattern / Return Early).
- Functions longer than 50 lines (extract complex logic into private helper functions prefixed with `_`).

Provide the fully refactored code and a bulleted list explaining exactly *why* you made each change.
```

# 3. The "Test-Verified Development" Prompt

**Use Case:** Generating rigorous unit tests for existing business logic without hitting real databases.

**Copy & Paste this into your AI Agent:**
```text
Act as a QA Automation Expert.

Read the service logic in `[INSERT FILE PATH]`.
Read our Testing Strategy in `27_TESTING_STRATEGY.md`.

Generate a comprehensive suite of unit tests using `pytest`.

Strict Constraints:
1. You MUST mock all database session calls (`AsyncSession`). Do not attempt to connect to a real database.
2. You MUST mock any external HTTP calls or LLM Gateway calls.
3. Test the "Happy Path" first.
4. Test at least two "Edge Cases" (e.g., entity not found, validation failure).
5. Test Tenant Isolation (ensure the service rejects an action if the `workspace_id` does not match).

Do not write tautological tests (e.g., `assert True == True`). Ensure your assertions actually verify the business logic.
```

# 4. The "AI Code Review" Prompt

**Use Case:** Having the AI act as a reviewer for your Pull Request before human eyes see it.

**Copy & Paste this into your AI Agent:**
```text
Act as the Chief Information Security Officer (CISO) and Lead Architect for this project.

Read the checklist in `42_CODE_REVIEW_CHECKLIST.md`.
Review the git diff of my current working tree.

I do not care about formatting (Black/Prettier handles that). I need you to look for fatal flaws:
1. SECURITY: Did I forget to filter a database query by `workspace_id`? Is there a risk of cross-tenant data leakage?
2. SECURITY: Are there any prompt injection vulnerabilities in how we pass user input to the LLM?
3. PERFORMANCE: Did I write a blocking synchronous call inside an `async def` function?
4. PERFORMANCE: Are there any N+1 query problems in my SQLAlchemy relationships?

If the code is flawless, just say "Approved." If you find flaws, quote the exact lines of code and explain the risk.
```

# 5. The "Frontend Component Generation" Prompt

**Use Case:** Building a new React component using our strict Next.js/Tailwind stack.

**Copy & Paste this into your AI Agent:**
```text
Act as a Staff Frontend Engineer specializing in Next.js (App Router) and Tailwind CSS.

I need to build a new UI component for the [INSERT MODULE NAME] module.
Before writing code, MUST read:
1. `22_CODING_STANDARDS.md` (TypeScript strictness)
2. `25_FRONTEND_GUIDELINES.md` (RSC vs Client components)

Task:
Build a [INSERT COMPONENT DESCRIPTION, e.g., "Account Profile Card"].

Constraints:
1. Use `shadcn/ui` primitives if applicable (assume they are in `components/ui/`).
2. Use Tailwind CSS for all styling via the `className` prop. Use the `cn()` utility for merging classes.
3. Make it a Server Component if it does not require interactivity. If it requires state/hooks, use `"use client"` at the absolute top of the file.
4. Define a strict TypeScript `interface` for the component props. No `any` types allowed.
5. Do not include data fetching logic inside this presentational component. Receive data via props.

Generate the clean, responsive code.
```

# 6. The "Zod Schema Sync" Prompt

**Use Case:** Translating a Backend Pydantic schema into a Frontend Zod schema for form validation.

**Copy & Paste this into your AI Agent:**
```text
Act as a Full-Stack Engineer bridging Python and TypeScript.

Here is a Python Pydantic schema from our backend:
```python
[PASTE PYDANTIC SCHEMA HERE]
```

Generate the corresponding TypeScript Zod schema and TypeScript inferred type.

Constraints:
1. The Zod schema MUST match the Pydantic validation rules exactly (e.g., if Pydantic has `max_length=50`, Zod must have `.max(50)`).
2. Export the inferred type as `[SchemaName]Data` (e.g., `export type AccountCreateData = z.infer<typeof AccountCreateSchema>;`).
3. Include descriptive error messages in the Zod validation (e.g., `.email("Must be a valid corporate email")`).
```
